为什么推荐使用notifyXxxChange方法更新RecyclerView？OC

发表于 2020-05-29 | 阅读次数
字数统计 2,275 | 阅读时长 9

前言

为什么我们在有数据刷新的时候推荐大家使用notifyItemChanged等方法而不使用notifyDataSetChanged方法呢？

源码分析

RecyclerView 中的观察者模式

  • Adapter 中的数据是被观察者
  • RecyclerViewDataObserver 是观察者，在数据集变化时做出对应的响应。
  • 注册过程：外部调用 RecyclerView#setAdapter 方法的时候，RecyclerView 会将自身的一个 RecyclerViewDataObserver 实例注册为监听者。
  • 通知过程：外部调用 Adapter#notifyXxxChanged 方法时，会触发 AdapterDataObservable#notifyXxxChanged，最终通知到上面注册的 RecyclerViewDataObserver 对应的方法。

以 notifyDataSetChanged 方法为例，最终会调用到RecyclerViewDataObserver#onChanged 方法，该方法 会处理结构性变化 。具体实现位于 processDataSetCompletelyChange 方法中，比较关键的是， RecyclerView#markKnownViewsInvalid

这两个标记与后续的更新以及缓存有很大的关系（在后面内容会说明）

假设此时没有 pendingUpdate（AdapterHelper#mPendingUpdates 中存储着外部调用 notifyXxx 方法触发的 op） ，会触发 requestLayout。 调用到 ViewRootImpl#requestLayout，触发 scheduleTraversals，等下一次垂直信号过来的时候，触发 performTraversal，重新进行布局。

调用到 RecyclerView#dispatchLayout 方法，触发 dispatchLayoutStep1、2、3。步骤 2 会触发 LayoutManager#onLayoutChildren （开启预测动画的情况下，步骤 1 也会触发 onLayoutChildren）

以 LinearLayoutManager 为例，onLayoutChildren 方法内部会调用 detachAndScrapAttachedViews 方法

也就是说，没有给 Adapter 设置 hasStableId 的情况下，notifyDataSetChange 方法触发的更新，会使 ViewHolder 被放到 Pool 中，而默认情况下，每种 itemType 的缓存最大个数为 5，屏幕的 itemType 比较单一，并且显示的个数较多，那么就会导致后续需要重新创建相应的 ViewHolder。

👆👆👆 划重点！

细粒度的更新方法，整体思路，某一个/多个 item 变化了，那么通知对应的 item ，诶，老铁，数据更新了，你得重新执行一次 bindViewHolder。

再来看看 notifyItemChange，最终也会调用到 RecyclerViewDataObserver#onItemRangeChanged ，首先触发 AdapterHelper#onItemRangeChanged，往队列中添加一个 update 操作。如果 mPendingUpdates 队列中只有一个 item，那么直接调用 RecyclerView#requestLayout ，监听下一个垂直同步信号，进行刷新

前面往队列中添加了一个 Op，在哪里取出来执行呢？

androidx.recyclerview.widget.AdapterHelper#consumePostponedUpdates

AdapterHelper 的构造函数接受一个 Callback 参数的时候，RecyclerView 在初始化时，会调用 RecyclerView#initAdapterManager，创建一个 AdapterHelper，此时会通过创建匿名内部类的方式构造一个 Callback 实例，传递给 AdapterHelper，AdapterHelper 在消费操作序列的时候，会调用 Callback 相应的方法

LayoutManager#onItemsUpdated(androidx.recyclerview.widget.RecyclerView, int, int, java.lang.Object)

  • AdapterHelper#consumeUpdatesInOnePass
    • androidx.recyclerview.widget.AdapterHelper.Callback#onDispatchSecondPass
    • androidx.recyclerview.widget.AdapterHelper.Callback#markViewHoldersUpdated
      • androidx.recyclerview.widget.RecyclerView#viewRangeUpdate
        • androidx.recyclerview.widget.RecyclerView#viewRangeUpdate
          • 给区间内的 Item 打上 ViewHolder.FLAG_UPDATE 标记，
          • RecyclerView.Recycler#viewRangeUpdate，同时也会给 RecyclerView.Recycler#mCachedViews 中的位于区间内的 ViewHolder 加上 ViewHolder.FLAG_UPDATE 标记，

细粒度的更新方法，

  • 会触发 到 AdapterHelper#onItemXxx方法，比如 AdapterHelper#onItemRangeChanged

    • 往队列中添加一个 UPDATE 操作。 mPendingUpdates.add(obtainUpdateOp(UpdateOp.UPDATE, positionStart, itemCount, payload));

      • 如果此时mPendingUpdates 长度为 1 ，则返回 true

        • 调用 RecyclerViewDataObserver#triggerUpdateProcessor

          • 根据情况，执行 mUpdateChildViewsRunnable 或者是 requestLayout

            void triggerUpdateProcessor() { if (POST_UPDATES_ON_ANIMATION && mHasFixedSize && mIsAttached) { ViewCompat.postOnAnimation(RecyclerView.this, mUpdateChildViewsRunnable); } else { mAdapterUpdateDuringMeasure = true; requestLayout(); }}

            • requestLayout 有什么作用？

                • AdapterHelper.Callback#onDispatchSecondPass
                  • AdapterHelper.Callback#dispatchUpdate

                • AdapterHelper.Callback#markViewHoldersUpdated
                  • 调用到 RecyclerView#viewRangeUpdate
                    • 遍历 ViewHolder ，添加 ViewHolder.FLAG_UPDATE 标志位。
                    • ViewHolder#addChangePayload
                    • 调用 RecyclerView.Recycler#viewRangeUpdate
                      • 如果 mCacheViews 中的 ViewHolder 在更新的范围内，给他们添加一个 ViewHolder.FLAG_UPDATE 标志位，并 将它们回收到 Pool 中？
                      • 貌似是添加到 mChangedScrap 中
                • AdapterHelper.Callback#offsetPositionsForAdd
                  • RecyclerView#offsetPositionRecordsForInsert
                    • ViewHolder#offsetPosition

注：结构变化和 item 变化

  • item 更改和 结构更改。item 更改是指单个 item 的数据已更新，但没有发生位置更改。结构更改是指在数据集 中 add、remove 或者是move item
    • 这个事件没有说明数据 集发生了什么变化，这使得任何观察者假定所有现有的 item 和结构都可能不再有效。LayoutManagers 将被迫完全重新绑定（ rebind） 和重新布局 （relayout)所有可见视图

  • 这么说 notifyDataSetChanged 和 notifyItemRangeChanged(0, dataSize) 有本质的差异？

  • notifyDataSetChanged

    • 有两类不同的数据变更事件，item 更改和 结构更改。item 更改是指单个 item 的数据已更新，但没有发生位置更改。结构更改是指在数据集 中 add、remove 或者是move item
      • 这个事件没有说明数据集发生了什么变化，这使得任何观察者假定所有现有的 item 和结构都可能不再有效。LayoutManagers 将被迫完全重新绑定（ rebind） 和重新布局 （relayout)所有可见视图
    • 对于 声明了 hasStableId 的 Adapter ，调用这个方法之后，RecyclerView 会尽量合成可见的 变化事件。
      • 这有助于实现动画和可视化对象的持久性，但是 每个独立的 item View 还是需要重新绑定和重新布局
    • 编写 Adapter 的时候，尽可能使用更具体的更改事件，这样可以收获更高的性能。notifyDataSetChanged() 可以作为兜底的方案。

  • 外部调用setAdapter 之后，recyclerView 会调用 RecyclerView.Adapter#registerAdapterDataObserver 注册为 AdapterData 的观察者（观察者实例是一个成员变量，直接 new 出来的），监听对应的事件

    • 具体实现 RecyclerViewDataObserver

      • 如果 notifyItemRangeChanged 会触发 RecyclerViewDataObserver#onChanged

        @Overridepublic void onChanged() { assertNotInLayoutOrScroll(null);//标记为结构性变化 mState.mStructureChanged = true;//处理完整的数据变化 processDataSetCompletelyChanged(true);//如果没有 延迟的更新，则调用 requestLayout 重新布局 if (!mAdapterHelper.hasPendingUpdates()) { requestLayout(); }}

        • 更新两个标志位，mDispatchItemsChangedEvent，用到 |= 运算符，仅当两个为false，结果才为 false。调用 RecyclerView#markKnownViewsInvalid 将 viewHolder 标记为无效的，在 mAdapter == null 或者是 !mAdapter.hasStableIds() 的情况下（一般不会去设置 hasStableIds 为 true，也就是大部分情况下），会清除 mCacheViews，将其添加到 RecyclerViewPool 中。

        • processDataSetCompletelyChanged 方法注释： 据我们所知，数据集已经完全变了。

          • 只要布局变化，所有已经 attach 的 item 都该被丢弃或者是执行动画。

            • 已经关联的 item 被标记为 invalid

          • 因为 在一个「数据集完全变化」事件和一个「布局变化」的事件之间， item 仍然可能被预获取，所以所有缓存的 item 都被丢弃了

            void processDataSetCompletelyChanged(boolean dispatchItemsChanged) { mDispatchItemsChangedEvent |= dispatchItemsChanged; mDataSetHasChangedAfterLayout = true; markKnownViewsInvalid();}

              • 应该不会，因为后续步骤还可能会回收当前 attached 在RV 上 的 ViewHolder
