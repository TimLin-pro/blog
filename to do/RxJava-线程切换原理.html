1.subscribeOn

包装为一个 ObservableSubscribeOn

ObservableSubscribeOn#subscribeActual

1、parent就是我们包装后的observer,其内部保存了下游的observer
2、source即通过ObservableSubscribeOn 包装后存储我们上游的observable

所以run里面的source.subscribe(parent);即为wrapper的observer订阅了上游的observable,触发了上游observable的subscribeActual，开始执行数据的分发

小结

因为订阅是从外层往内传的，所以多次设置subscribeOn()时，只有第一次生效。

  • 业务代码中，观察者订阅的时候。 observable.subscribe触发
  • —>(Wrapper)Observable.subscribeActual()内部调用
  • —>parent.setDisposable(scheduler.scheduleDirect(new SubscribeTask(parent)));,
  • –>scheduler在指定线程调用(完成线程切换)
  • –>SubscribeTask.run,run内部调用
  • –>原Observable.subscribe( (Wrapper)Observer )方法，触发
  • –>(原)Observable.subscribeActual()开始数据分发
  • 此时分发给的是(Wrapper)Observer，那应该是(Wrapper)Observer分发给了(原）Observer

scheduler 具体的调度过程：

  • scheduler.scheduleDirect无延迟调用
  • ->worker->worker调度->DisposeTask
  • ->DisposeTask.run执行->decoratedRun.run();
  • decoratedRun即我们外部的SubscribeTask

2.observeOn

通过中间增加一个Observable和Observer来实现线程的切换。

io.reactivex.Observable#observeOn(io.reactivex.Scheduler)

Observable#observeOn(Scheduler, boolean, int)

observeOn 方法会返回一个 新创建的 ObservableObserveOn，

ObservableObserveOn<T> 继承自 AbstractObservableWithUpstream<T, T>

而 AbstractObservableWithUpstream 又继承自 Observable

所以 新创建的ObservableObserveOn 类可以看做是 对 Observable 的包装。

根据Java 的「多态」特性，当 subscribe 方法被调用之后，会触发回调 subscribeActual 方法

ObservableObserveOn#subscribeActual

首先判断 scheduler 是否TrampolineScheduler 的子类，如果是，则会在当前线程调度。

否则，创建对应的 worker ，然后将订阅者包装为 ObserveOnObserver。

也就是说 observeOn 对上游的 Observable 和下游的 Observer 都包装了一次。

上游的 Observable 包装为 ObservableObserveOn

下游的 Observer 包装为 ObserveOnObserver

当下游调用了 subscribe 方法时，触发 —> subscribeActual —>包装了下游的 Observer。

Worker 与 Scheduler

为什么 Scheduler 不定义为一个抽象类

  • 【第一、二点】Java 在 8.0 版本之前，接口不支持默认实现方法，而 Scheduler 需要多个方法提供默认实现。RxJava 考虑到兼容性，将长期使用低版本的 Java 。
  • 【第三、四点】如果将 Scheduler 定义为接口，那么需要添加一个 AbstractScheduler 抽象类，实现接口的默认方法实现。

Scheduler 提供方法如下 ：

  • #createWorker() 抽象方法 ：创建 Worker 。
  • #now() 默认方法 ：返回当前时间。
